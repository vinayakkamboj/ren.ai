## What Nucode is

Nucode is an autonomous AI coding agent that generates and edits React applications integrating the **Nutrient Web SDK**. Not docs lookup, not generic coding, not multi-product theory — its job is to write correct, working Nutrient Web SDK code and build complete demo apps from a plain English description.

Nucode runs Claude with a Web SDK–first system prompt and the full Nutrient documentation reference embedded as context. It answers technical questions, generates production-quality code, and can do both in the same response.

## Architecture

### Host stack

| Layer | Technology |
|---|---|
| Host app | Next.js 15 (App Router), React 18, TypeScript |
| Preview sandbox | Sandpack (`@codesandbox/sandpack-react`) — in-browser Vite+React runtime |
| AI models | Nucode model tiers — user-selectable per workspace (see "Model tiers" section); Nucode Spark always powers plan/design/classify/ask |
| AI SDK | Vercel AI SDK v4 (`ai`, `@ai-sdk/anthropic`) |
| Database | Supabase (project files, chat sessions, workspaces) |

### How Nucode finds the right code

**Repository intelligence scoring** (`features/ai/repository-intelligence.ts`)

Before every response, Nucode scores every file in the project to decide which ones are most relevant to the current request. The scoring factors are:

- `NUTRIENTWEBBUILDER.md` → +25 (always relevant — project memory)
- `src/App.tsx` → +24 (always relevant — root entry point)
- Files matching request tokens (domain words extracted from the user message) → +8 per match
- Files in the error stack trace → +40 (runtime errors force-include the crashing file)
- Recent changed files → +16 (likely still relevant from the last edit)

The top-scoring files are packed into the context window in priority order. Files that would exceed the token budget are truncated at safe line limits (pages: 200 lines, components: 150 lines, hooks/services: 100 lines, data: 60 lines).

**Context pack format** (`features/ai/repository-intelligence.ts → formatContextPackForPrompt`)

The packed files are sent as a structured markdown section so Nucode sees the project's current state without hallucinating missing imports.

### How Nucode writes files

Nucode outputs a `<file_patches>` JSON block:

```json
{
  "plan": "one sentence summary of what changed",
  "changes": [
    { "path": "src/App.tsx", "content": "FULL FILE CONTENT" },
    { "path": "src/pages/DocumentPage.tsx", "content": "FULL FILE CONTENT" }
  ],
  "deletes": ["src/old-component.tsx"],
  "renames": [{ "from": "src/Old.tsx", "to": "src/New.tsx" }]
}
```

This is parsed by `parseFilePatchPlan` with five recovery strategies for truncated AI output (direct parse, suffix repair, boundary truncation, character scanning, XML-tag fallback).

Files are applied **atomically** — the entire candidate is validated for fatal issues before any file is written to the project.

### Fatal issue detection

Before applying any generated file set, `detectFatalIssues` checks for:

| Issue type | Description |
|---|---|
| `missing-app-tsx` | No `src/App.tsx` in the patch plan |
| `truncated` | A file is suspiciously short (< 200 chars for key files) — indicates AI truncation |
| `sdk-npm-import` | Any file contains `import ... from "@nutrient-sdk/viewer"` (SDK must always be CDN-only) |
| `missing-file` | A relative import in one file has no matching file in the patch plan |

If a fatal issue is detected, the candidate is rejected and the full-build pipeline triggers a second pass.

### CDN-only rule (critical)

The Nutrient Web SDK is **never** imported from npm. The 5 MB bundle breaks Sandpack's in-browser bundler.

`src/NutrientViewer.tsx` is the canonical wrapper injected into every preview. It loads the SDK at runtime from Nutrient's CDN as `window.NutrientViewer`. Generated app code always uses:

```tsx
import { NutrientViewer } from "./NutrientViewer";  // from App.tsx
import { NutrientViewer } from "../NutrientViewer"; // from pages/ or components/
```

### How Nucode picks brand names and colors

**Brand names** are generated by the plan phase (Nucode Spark, `buildLightPlanPrompt`). The rule is: no generic suffix-based SaaS names ("FlowHub", "CoreDesk"). The planner is explicitly told to pick evocative, product-specific names — examples it is shown: "Northwind", "Lumen", "Cartograph", "Anvil", "Atlas", "Foreman".

**Color palettes** are generated by the design phase (`buildDesignSpecPrompt`). Every palette token is output as space-separated HSL values (e.g. `170 82% 39%`) so Tailwind opacity modifiers work (`bg-primary/10`). The anti-blue rule is enforced: `#4f8cff`, `#38bdf8`, `#0ea5e9`, `#2563eb`, `#3b82f6` are forbidden as defaults. Every product gets a palette derived from its emotional register (kinetic orange for construction, clinical sage for healthcare, premium plum for legal, etc.).

## Pipelines

### History

The agent originally shipped with **seven pipelines** (`chat`, `light-build`, `full-build`, `iterative`, `sdk-focused`, `error-fix`, `generic`) driven by a Haiku classifier. This over-segmentation caused crashes: the classifier too often routed new projects to `full-build`, which suffered from AI truncation on large files — the Sandpack sandbox recovered only partial output, breaking `App.tsx` imports at runtime.

**What was removed:** all seven legacy pipelines and the classifier-driven routing. A hard override (`!hasExistingBuild → full-build`) that was masking the real issue was also removed.

**Why simplification worked:** `light-build` was succeeding on ~90% of requests. The problem was forced routing, not the pipeline logic itself. Removing the override and making `light` the permanent default restored stability immediately.

---

### Current pipelines (3)

| Pipeline | Best for | Phases |
|---|---|---|
| **Light** | All new builds, edits, and fixes — the default | plan → design → build |
| **Full** | Complex multi-page first builds | plan → design → build → (pass 2 if needed) |
| **Deep** | Surgical Nutrient Web SDK coding + expert-level changes | single pass, full SDK context |

**Light** (default): Nucode Spark for plan + design, the user-selected model tier for the build pass. Produces 5–7 files in one response. Smart 2-pass is available when needed (missing App.tsx, truncated files, < 3 files generated).

**Full**: Same as Light but with a quality-assurance reviewer pass after build. Used for complex multi-page first builds where the extra pass is worth the latency.

**Deep**: User-selected model tier, single pass, 16k token output budget. Full Nutrient Web SDK docs embedded as context. Designed to answer questions and generate code in the same response. This is the Web SDK expert lane. Internal pipeline type: `"deep"`. Nucode is the app/agent name, not the third pipeline name.

## Model tiers

Nucode ships its own model lineup. Users pick the model per workspace from the **Model** dropdown in the chat panel; the choice applies to the heavy code-generation phases (Light/Full **build** pass and **Deep**). Brand names are Nucode-owned — the underlying provider model is resolved server-side and never shown to end users.

| Nucode tier | Usage level | Underlying model | Best for |
|---|---|---|---|
| **Nucode Spark** | Low | `claude-haiku-4-5-20251001` | Fastest/cheapest — simple edits and quick fixes. Also powers ALL plan, design, classify, and ask phases regardless of the user's selection. |
| **Nucode Flow** | Medium | `claude-sonnet-4-6` | **Default tier.** Balanced speed and quality for everyday builds. |
| **Nucode Forge** | High | `claude-opus-4-8` | Heavy builds — near-top quality at half Apex cost. |
| **Nucode Apex** | Max | `claude-fable-5` | Maximum quality, highest token usage (~30% more tokens per request due to its tokenizer, plus 2× Forge pricing). |

### Wiring

- **Single source of truth:** `features/ai/model-registry.ts` (`MODEL_TIERS`, `resolveModelTier`). Add or change tiers only there — the chat UI, API route, and admin dashboard all read from it.
- **Client:** `components/workspace/ChatPanel.tsx` renders the dropdown (`brandName · usageLevel`), persists the choice per workspace in localStorage (`nucode-model-tier-<workspaceId>`), and sends `modelTier` in the chat request body.
- **Server:** `app/api/ai/chat/route.ts` resolves the tier via `getAgentModel(modelTier)` for build/deep. Invalid or missing tiers fall back to Flow. The `ANTHROPIC_MODEL` env var is an ops override used only when no tier is sent.
- **Tracking:** every request logs to `token_usage` against the **resolved** model ID, so per-model burn is visible in the usage dashboard. The admin Pipelines page (`/admin/pipelines`) shows the full tier → model mapping with usage levels and the current default.
- **Verified:** all four underlying models confirmed live against the Anthropic API (2026-06-12).

## Future MCP integration

This section documents the planned MCP architecture. Nothing below is live today — these are connection points being designed for a future release.

### Architecture intent

Nucode currently embeds a static snapshot of Nutrient documentation as its system prompt context. This works well but has two limitations: the snapshot goes stale between releases, and it can't fetch live document state from a running Document Engine or DWS instance.

MCP (Model Context Protocol) integration will replace static context injection with live tool calls. Nucode will load enabled tools from `NUCODE_MCP_REGISTRY` at the start of each session.

### Planned tools

| Tool ID | Name | Purpose |
|---|---|---|
| `nutrient-docs-search` | Nutrient Docs Search | Semantic search over `docs.nutrient.io` — live, version-accurate answers |
| `nutrient-code-examples` | Nutrient Code Examples | Fetch runnable Web SDK examples by feature |
| `nutrient-changelog` | Nutrient Changelog | Query SDK changelogs — version-accurate upgrade paths |
| `dws-api-tools` | DWS Processor API Tools | Live DWS API calls for OCR, conversion, redaction, watermarking |
| `document-engine-api` | Document Engine REST API | Query a connected Document Engine instance |
| `internal-docs` | Nutrient Internal Docs | Access internal Confluence/Notion pages and engineering RFCs |

### Admin panel

The Admin panel at `/admin/pipelines` (restricted to `vinayak.kamboj@nutrient.io`) shows all pipelines with their phase flow, per-phase model/token config, input/output tags, and MCP hook slots. Each hook slot shows its current status (Planned / Connected / Disabled). This is the surface where MCP tools will be wired in when the integration ships.

### Connection model

Each pipeline phase in `features/ai/pipeline-registry.ts` has an `mcpHooks` array:

```ts
mcpHooks: [
  {
    id: "nutrient-docs-search",
    name: "Nutrient Docs Search",
    triggerOn: "before-prompt",
    status: "planned",
  }
]
```

When a tool moves from `"planned"` to `"connected"`, the phase will call it before or after the LLM step and inject the result into the prompt.

## Hallucination rule

Nucode must not invent Nutrient API method signatures, event names, or config properties. If the exact syntax for a feature is not in the embedded context, it must say so and point to the relevant source:

> "I'm not certain of the exact API for X. Check https://www.nutrient.io/api/web/ or https://www.nutrient.io/guides/web/[feature]/ for the correct signature."

This is especially important for rarely-used toolbar item types, advanced annotation properties, and Document Engine REST endpoints.
